3.1 Проблемы плохой структуры данных
Аномалии в ненормализованной таблице
Рассмотрим таблицу для хранения заказов:

order_id	customer_name	customer_phone	product_id	product_name	product_price
101	Иван Иванов	123-45-67	1	Ноутбук	75000
101	Иван Иванов	123-45-67	2	Мышь	1500
102	Иван Иванов	123-45-67	3	Клавиатура	3500
103	Мария Петрова	987-65-43	1	Ноутбук	75000
Аномалии:

Аномалия вставки: Нельзя добавить нового клиента без заказа

Аномалия обновления: Чтобы изменить телефон Ивана, нужно менять в трех строках

Аномалия удаления: При удалении заказа 103 потеряем информацию о клиенте Мария

3.2 Нормализация
Первая нормальная форма (1НФ)
Требование: Каждый атрибут должен содержать атомарные значения (не массивы и не составные значения).

❌ Нарушение:

id	name	phones
1	Иван	12345, 67890
✅ 1НФ:

id	name	phone
1	Иван	12345
1	Иван	67890
Вторая нормальная форма (2НФ)
Требование: Находиться в 1НФ и не иметь частичных зависимостей от составного ключа.

❌ Нарушение (если ключ составной order_id + product_id):

order_id	product_id	product_name	quantity
101	1	Ноутбук	1
Проблема: product_name зависит только от product_id, не от order_id

✅ 2НФ:

sql
-- Таблица order_items
| order_id | product_id | quantity |
|----------|------------|----------|
| 101      | 1          | 1        |

-- Таблица products
| product_id | product_name |
|------------|--------------|
| 1          | Ноутбук      |
Третья нормальная форма (3НФ)
Требование: Находиться в 2НФ и не иметь транзитивных зависимостей (неключевые атрибуты не должны зависеть от других неключевых).

❌ Нарушение:

order_id	customer_id	customer_city	customer_phone
101	1	Москва	12345
Проблема: customer_city зависит от customer_id, а не от order_id

✅ 3НФ:

sql
-- Таблица orders
| order_id | customer_id |
|----------|-------------|
| 101      | 1           |

-- Таблица customers
| customer_id | customer_city | customer_phone |
|-------------|---------------|----------------|
| 1           | Москва        | 12345          |
Денормализация: когда можно нарушить правила
Причины для денормализации:

Повышение производительности (меньше JOIN'ов)

Хранение предвычисленных агрегатов

Работа с аналитическими (OLAP) системами

Пример денормализации:

sql
-- Добавляем вычисляемое поле для ускорения запросов
ALTER TABLE orders ADD COLUMN item_count INT;

-- Триггер будет обновлять это поле при изменении позиций заказа
3.3 Типы связей между таблицами
Связь "Один-к-одному" (1:1)
Редкая связь, обычно для разделения часто/редко используемых полей.

ascii
┌─────────────┐          ┌─────────────────┐
│ users       │          │ user_profiles   │
├─────────────┤          ├─────────────────┤
│ id          │◄────────►│ user_id (FK)    │
│ login       │          │ bio             │
│ password    │          │ avatar_url      │
└─────────────┘          └─────────────────┘
Связь "Один-ко-многим" (1:M)
Наиболее распространенная связь.

ascii
┌─────────────┐          ┌─────────────────┐
│ categories  │          │ products        │
├─────────────┤          ├─────────────────┤
│ id (PK)     │◄─────────┤ category_id (FK)│
│ name        │          │ name            │
└─────────────┘    М     │ price           │
                     Н    └─────────────────┘
                      О   
                       Г  -- Одна категория → много товаров
                        И 
                         М
Связь "Многие-ко-многим" (M:M)
Реализуется через промежуточную таблицу.

ascii
┌─────────────┐      ┌─────────────────┐      ┌─────────────┐
│ students    │      │ student_courses │      │ courses     │
├─────────────┤      ├─────────────────┤      ├─────────────┤
│ id (PK)     │◄────►│ student_id (FK) │      │ id (PK)     │
│ name        │      │ course_id (FK)  │◄────►│ title       │
└─────────────┘      └─────────────────┘      └─────────────┘
      │                      ▲                        ▲
      └──────────────────────┘                        │
          Студент может быть на многих курсах         │
                                                      │
      ┌──────────────────────────────────────────────┘
      └── На курсе может быть много студентов
3.4 Ссылочная целостность и внешние ключи
Создание таблиц с внешними ключами
sql
CREATE TABLE departments (
    id INT PRIMARY KEY,
    name VARCHAR(100) NOT NULL
);

CREATE TABLE employees (
    id INT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    department_id INT,
    FOREIGN KEY (department_id) 
        REFERENCES departments(id)
        ON DELETE SET NULL   -- при удалении отдела ставим NULL
        ON UPDATE CASCADE    -- при изменении id отдела обновляем ссылку
);

-- Альтернативные варианты поведения:
-- ON DELETE RESTRICT  — запретить удаление, если есть связанные записи
-- ON DELETE CASCADE   — удалить все связанные записи
-- ON DELETE SET NULL  — установить NULL в связанных записях
-- ON DELETE SET DEFAULT — установить значение по умолчанию
-- ON DELETE NO ACTION  — аналогично RESTRICT
3.5 Соединение таблиц (JOIN)
Типы JOIN с визуализацией
ascii
Исходные таблицы:

Таблица A (employees):        Таблица B (departments):
┌────┬─────────┬────────┐    ┌────┬──────────────┐
│ id │ name    │ dept_id│    │ id │ dept_name    │
├────┼─────────┼────────┤    ├────┼──────────────┤
│ 1  │ Иван    │ 10     │    │ 10 │ IT           │
│ 2  │ Мария   │ 20     │    │ 20 │ HR           │
│ 3  │ Петр    │ NULL   │    │ 30 │ Sales        │
│ 4  │ Ольга   │ 10     │    └────┴──────────────┘
└────┴─────────┴────────┘

INNER JOIN: только совпадения
┌─────────┬──────────────┐
│ name    │ dept_name    │
├─────────┼──────────────┤
│ Иван    │ IT           │
│ Мария   │ HR           │
│ Ольга   │ IT           │
└─────────┴──────────────┘
(Петр не попал, Sales не попали)

LEFT JOIN: все из A + совпадения из B
┌─────────┬──────────────┐
│ name    │ dept_name    │
├─────────┼──────────────┤
│ Иван    │ IT           │
│ Мария   │ HR           │
│ Петр    │ NULL         │
│ Ольга   │ IT           │
└─────────┴──────────────┘

RIGHT JOIN: все из B + совпадения из A
┌─────────┬──────────────┐
│ name    │ dept_name    │
├─────────┼──────────────┤
│ Иван    │ IT           │
│ Мария   │ HR           │
│ Ольга   │ IT           │
│ NULL    │ Sales        │
└─────────┴──────────────┘

FULL OUTER JOIN: все отовсюду
┌─────────┬──────────────┐
│ name    │ dept_name    │
├─────────┼──────────────┤
│ Иван    │ IT           │

│ Ольга   │ IT           │

└─────────┴──────────────┘
Примеры JOIN на практике
sql
-- INNER JOIN: только сотрудники с отделами
SELECT e.name, d.dept_name
FROM employees e
INNER JOIN departments d ON e.department_id = d.id;

-- LEFT JOIN: все сотрудники, включая тех, у кого нет отдела
SELECT e.name, d.dept_name
FROM employees e
LEFT JOIN departments d ON e.department_id = d.id;

-- JOIN с дополнительными условиями
SELECT o.id, c.name, o.total_amount
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE o.total_amount > 10000
ORDER BY o.total_amount DESC;

-- Множественные JOIN
SELECT 
    o.id as order_id,
    c.name as customer,
    p.name as product,
    oi.quantity
FROM orders o
JOIN customers c ON o.customer_id = c.id
JOIN order_items oi ON o.id = oi.order_id
JOIN products p ON oi.product_id = p.id;
